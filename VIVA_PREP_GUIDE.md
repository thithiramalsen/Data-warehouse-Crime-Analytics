## 3) SCD Types: Used vs Not Used

## What is used in this repo

1. SCD Type 2 structure (implemented structurally for DimArea)
- Evidence:
  - `valid_from`, `valid_to`, `is_current` are present and populated in DimArea flow.
  - Fact lookup metadata includes these columns from DimArea.
- Practical interpretation:
  - Your model is Type-2-ready and stores temporal attributes.
  - In the current package version, it mainly acts like initial snapshot loading for DimArea unless extra change-detection/update logic is added.

2. Accumulating snapshot concept (fact-side, not an SCD type)
- `LoadAccumulating.dtsx` updates completion timestamp and process duration over time.

## What is not used (in current main package set)

1. Full SCD Type 2 change handling pipeline
- No explicit expire-current-row (`is_current = 0`, close `valid_to`) + insert-new-version logic found in current `LoadDimensions.dtsx`.

2. SCD Type 1
- No overwrite update path for changed dimension attributes (no direct UPDATE/merge for Type 1 behavior in current dimensions package).

3. SCD Type 3
- No previous/current paired columns (for example previous division, previous description) in the shown dimension loads.

How to explain this in viva:
- "I implemented a Type-2-capable schema for DimArea with effective dating and current-row flag. In this submission, the ETL is focused on clean load and conformance; full Type-2 change detection/row versioning update path can be added as next enhancement."

## 5) Common Viva Questions (Strong Answers)

1. Why staging first?
- To isolate raw ingestion from business logic, enable repeatable loads, and simplify debugging.

2. Why surrogate keys in fact table?
- Stable joins, better historical modeling, and faster dimensional analytics.

3. Why did you include `valid_from`, `valid_to`, `is_current`?
- To support temporal analysis and future Type 2 version tracking.

4. Which SCD types did you implement?
- Type 2 structural support in DimArea; full Type 2 change detection/update and Type 1/Type 3 are not fully implemented in the current main ETL package.

5. Why accumulating snapshot update?
- To calculate lifecycle/process KPIs like completion time and duration without rewriting the whole fact load.
